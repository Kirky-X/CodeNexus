## ADDED Requirements

### Requirement: 提取器 FQN 格式一致性

Python、TypeScript、C、Fortran 四个语言提取器 SHALL 使用 `FqnGenerator::generate(project, file_path, name, language)` 生成节点 id（qualified_name），产出 spec 合规的点分格式 `project.dir.file.name`。提取器 MUST NOT 使用 `format!("{file_path}::{name}")` 格式生成节点 id。

#### Scenario: Python 提取器生成点分 FQN

- **WHEN** `PythonExtractor::extract` 解析包含 `def foo():` 的 Python 源码，传入 `project="proj"`、`file_path="/tmp/demo/main.py"`
- **THEN** 生成的 Function 节点 `id` 形如 `proj.tmp.demo.main.foo`（点分格式），与 `FqnGenerator::generate` 输出一致

#### Scenario: TypeScript 提取器生成点分 FQN

- **WHEN** `TypeScriptExtractor::extract` 解析包含 `function bar():` 的 TypeScript 源码，传入 `project="proj"`、`file_path="/tmp/demo/main.ts"`
- **THEN** 生成的 Function 节点 `id` 形如 `proj.tmp.demo.main.bar`（点分格式），与 `FqnGenerator::generate` 输出一致

#### Scenario: C 提取器生成点分 FQN

- **WHEN** `CExtractor::extract` 解析包含 `int baz() {` 的 C 源码，传入 `project="proj"`、`file_path="/tmp/demo/main.c"`
- **THEN** 生成的 Function 节点 `id` 形如 `proj.tmp.demo.main.baz`（点分格式），与 `FqnGenerator::generate` 输出一致

#### Scenario: Fortran 提取器生成点分 FQN

- **WHEN** `FortranExtractor::extract` 解析包含 `subroutine qux` 的 Fortran 源码，传入 `project="proj"`、`file_path="/tmp/demo/main.f90"`
- **THEN** 生成的 Function 节点 `id` 形如 `proj.tmp.demo.main.qux`（点分格式），与 `FqnGenerator::generate` 输出一致

#### Scenario: 所有定义节点类型使用点分 FQN

- **WHEN** 任一提取器（Python/TypeScript/C/Fortran）解析源码，生成 Function/Struct/Enum/Class/Module/Namespace/GlobalVar/Typedef/Const/Static 等任一 label 的节点
- **THEN** 该节点的 `id` 字段 MUST 是 `FqnGenerator::generate` 的输出，点分格式

### Requirement: CALLS 边生成完整性

Python、TypeScript、C、Fortran 四个语言提取器的 `extract_call` 函数 SHALL 接收 `current_func: Option<&str>` 参数，并通过 `make_qn(file_path, current_func, project)` 生成 `CallInfo.caller_qn`。当调用发生在函数体内时，`caller_qn` MUST NOT 为 `None`，且 MUST 与该函数定义节点的 `id` 匹配。

#### Scenario: Python 函数内调用生成非 None caller_qn

- **WHEN** `PythonExtractor::extract` 解析包含以下源码：
  ```python
  def caller():
      callee()
  ```
- **THEN** 生成的 `CallInfo` 记录中 `caller_qn` 是 `Some("proj.tmp.demo.main.caller")`（点分格式），`callee_name` 是 `"callee"`

#### Scenario: TypeScript 函数内调用生成非 None caller_qn

- **WHEN** `TypeScriptExtractor::extract` 解析包含以下源码：
  ```typescript
  function caller(): void {
      callee();
  }
  ```
- **THEN** 生成的 `CallInfo` 记录中 `caller_qn` 是 `Some("proj.tmp.demo.main.caller")`（点分格式），`callee_name` 是 `"callee"`

#### Scenario: C 函数内调用生成非 None caller_qn

- **WHEN** `CExtractor::extract` 解析包含以下源码：
  ```c
  int caller(void) {
      callee();
      return 0;
  }
  ```
- **THEN** 生成的 `CallInfo` 记录中 `caller_qn` 是 `Some("proj.tmp.demo.main.caller")`（点分格式），`callee_name` 是 `"callee"`

#### Scenario: Fortran 函数内调用生成非 None caller_qn

- **WHEN** `FortranExtractor::extract` 解析包含以下源码：
  ```fortran
  subroutine caller
      call callee
  end subroutine caller
  ```
- **THEN** 生成的 `CallInfo` 记录中 `caller_qn` 是 `Some("proj.tmp.demo.main.caller")`（点分格式），`callee_name` 是 `"callee"`

#### Scenario: 顶层调用（无函数上下文）caller_qn 为 None

- **WHEN** 任一提取器解析的调用表达式不在任何函数体内（如模块级初始化代码）
- **THEN** 生成的 `CallInfo` 记录中 `caller_qn` 是 `None`（这是合法行为，与 rust_extractor 一致）

#### Scenario: CALLS 边端到端生成

- **WHEN** 对包含函数 A 调用函数 B 的源码执行完整索引流程（extract → resolve → load）
- **THEN** 数据库中 MUST 存在一条 CALLS 边，其 `source` 是函数 A 的节点 id，`target` 是函数 B 的节点 id，且 `source`/`target` 值能与 `find_symbol_node_ids` 查询到的节点 id 匹配

### Requirement: C 提取器 reads/writes 提取

C 语言提取器 SHALL 实现 reads（BR-TRACE-005）和 writes（BR-TRACE-006）提取，移除 `src/parse/c.rs` 第 53 行的 TODO 标记。读取和写入记录 MUST 在函数体内捕获，`reader_qn`/`writer_qn` MUST 是包含函数的点分 FQN。

#### Scenario: C 函数内 identifier 读取提取

- **WHEN** `CExtractor::extract` 解析包含以下源码：
  ```c
  int caller(int x) {
      return x;
  }
  ```
- **THEN** 生成的 `ReadInfo` 记录中 `reader_qn` 是 `Some("proj.tmp.demo.main.caller")`，`var_name` 是 `"x"`

#### Scenario: C 函数内 init_declarator 写入提取

- **WHEN** `CExtractor::extract` 解析包含以下源码：
  ```c
  void caller(void) {
      int y = 1;
  }
  ```
- **THEN** 生成的 `WriteInfo` 记录中 `writer_qn` 是 `Some("proj.tmp.demo.main.caller")`，`target_name` 是 `"y"`

#### Scenario: C 函数内 assignment_expression 写入提取

- **WHEN** `CExtractor::extract` 解析包含以下源码：
  ```c
  void caller(void) {
      int y;
      y = 2;
  }
  ```
- **THEN** 生成的 `WriteInfo` 记录中 `writer_qn` 是 `Some("proj.tmp.demo.main.caller")`，`target_name` 是 `"y"`

#### Scenario: C 声明位置的 identifier 不被误识别为读取

- **WHEN** `CExtractor::extract` 解析 `int x = 1;`（`x` 在 declarator 位置）
- **THEN** `x` MUST NOT 出现在 `ReadInfo` 记录中（它应作为 WriteInfo 的 target_name，而非 ReadInfo 的 var_name）

#### Scenario: C 顶层声明的 identifier 不生成读写记录

- **WHEN** `CExtractor::extract` 解析 `int g = 0;`（顶层全局变量声明，不在函数体内）
- **THEN** 不生成 `ReadInfo` 或 `WriteInfo` 记录（`current_func` 为 `None`，读写提取只在函数体内进行）

### Requirement: Task 6 遗留测试修复

本变更 SHALL 修复 `fix-id-fqn-consistency` spec Task 6 遗留的 11 个测试失败，使 `cargo test --lib` 全绿。测试断言 MUST 与源码实际行为一致。

#### Scenario: loader.rs node_to_row 列数断言与实现一致

- **WHEN** 运行 `src/storage/loader.rs` 的 `node_to_row_*_has_*_columns` 系列测试
- **THEN** 所有列数断言 MUST 与 `node_to_row` 函数的实际输出列数一致（Class/Struct/Enum/Trait=11，Const=10，Parameter=9，Static=9，TypeAlias=9，Typedef=8）

#### Scenario: rust_extractor.rs pattern 测试断言与实现一致

- **WHEN** 运行 `src/parse/rust_extractor.rs` 的 `tuple_destructuring_pattern` 和 `struct_pattern` 测试
- **THEN** `tuple_destructuring_pattern` 测试断言验证提取首个绑定 "a"（非 "a" 和 "b"）；`struct_pattern` 测试断言验证提取类型名 "P"（非字段名 "x"）

#### Scenario: 全量测试通过

- **WHEN** 运行 `cargo test --lib`
- **THEN** 所有测试 MUST 通过，无失败用例
